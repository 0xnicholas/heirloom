## 企业智能体的语义层

Heirloom 让 AI 智能体理解业务、通过受治理的工作流执行操作、持久化状态、解释所发生的一切，并始终处于硬系统边界之内。

将你的运营建模为语义本体：资源、关系、能力、状态机和操作。智能体查询当前与历史状态，调用确定性的业务操作，并将每一次尝试记录到可审计的事件日志中。

## 访问难题

今天，企业 AI 只有两个同样糟糕的选择：

- **沙箱模式** —— 智能体什么都看不到。安全，但无用。
- **数据库模式** —— 智能体拿到数据库连接串。灵活，但危险。

在第二种模式下，LLM 的幻觉可能变成一次 `DROP TABLE`。提示词里写"小心操作"只是请求，不是约束。缺失的环节是原始数据与智能体之间的语义层：一个操作被类型化、能力被显式声明、不安全请求被系统本身拒绝的层。

Heirloom 就是这个层。

## 九个受治理的智能体操作原语

智能体从这些构建块中获得持久的业务上下文。每个原语都在本体中定义，并由运行时强制执行。

### 资源（Resources）

具有稳定标识和类型化属性的事物。客户、订单、合同、资产、策略：智能体不能混淆的名词。

### 资源类型（Resource Types）

资源的语义契约：字段、能力、状态机与关系。在 Schema Registry 中注册并版本化。

### 能力（Abilities）

类型级能力契约。资源类型声明其实例能否被查询、修改、转移、复制、冻结或删除。如果未声明 `drop`，则任何角色——包括管理员——都无法创建删除该资源的操作。

### 关系（Relationships）

资源之间类型化的精确连接：

- **所有权（Ownership）** —— 生命周期依赖；删除时级联。
- **引用（Reference）** —— 无依赖的指针；被引用方删除时引用断裂。
- **关联（Association）** —— 独立的链接。

### 状态机（State Machines）

合法状态转移被预先声明。试图通过未定义的边移动资源会在类型层被拒绝，而不是靠手写的 `if` 语句。

### 操作（Actions）

唯一的结构化写入路径。每个操作都流经九步流水线：认证、角色、能力、门禁、状态、校验、执行、事件、通知。

### 函数（Functions）

资源上的只读计算。函数可被智能体、操作或应用调用，但从不改变资源状态。

### 角色（Roles）

在本体、资源类型或资源实例范围内授予的静态身份。角色将行为体连接到能力。

### 能力票据（Capabilities）

从角色派生的短期、可撤销票据。能力票据指定行为体可以对哪些资源执行哪些操作。

## 从模型到智能体工作流

一个典型的供应商认证工作流，带有记忆、规则与可审计性：

1. **定义业务上下文** —— 建模 `Vendor`、`Certificate` 及它们之间的关系。
2. **声明规则** —— 一个约束将证书过期的活跃供应商标记为违规。
3. **让智能体在闭环内工作** —— 运行时持续评估规则。当证书过期时，违规浮出水面，任务被路由给智能体，智能体收集上下文、请求续期并记录变更。

```mermaid
graph LR
    A[证书过期] --> B[创建违规]
    B --> C[任务路由给智能体]
    C --> D[智能体请求续期]
    D --> E[记录新证书]
    E --> A
```

## 可靠智能体之下的层

Heirloom 不是模型。它是模型之下的运营基底。

| 支柱 | 作用 |
|------|------|
| **Schema Registry** | 存储资源类型、能力、状态机与角色。智能体的运行时字典。 |
| **Mapping Engine** | 将业务字段映射到物理数据源。智能体查询 `Customer.tier`，而非表列。 |
| **Query Resolver** | 将 JSON DSL 查询转换为跨多源的执行计划。 |
| **Perspective Engine** | 根据调用者角色裁剪返回的字段与关系。 |
| **Event Log** | 以不可变事件的形式追加每一次操作，无论成功或被拒。 |

## 智能体工作成为可观察状态

工作流在记忆、规则、操作、审批与反馈中运行。

- **记忆** —— 每次智能体操作都会写下事实：它知道什么、改变了什么、使用了哪个来源、谁批准了、何时发生。
- **治理** —— 规则、工具边界与审批都是显式的。智能体在业务流程内部运作。
- **修正** —— 失败成为可检查的状态。修正成为新的事实。下一次运行拥有更好的上下文，而无需抹除历史。

## 建模、查询、执行、观察、修正

1. **建模** 运营图 —— 实体、关系与规则。
2. **查询** 图 —— 使用结构化 JSON DSL。
3. **执行** —— 通过具有类型边界的受治理操作。
4. **观察** —— 在 Event Log 中查看每一次尝试。
5. **修正** —— 修正模型或数据，让下一次运行继承上下文。

## 从一个智能体工作流开始

编码你的智能体所需的业务上下文、规则、工具、审批与反馈循环。Heirloom 让这个工作流变得安全、可观察且可复用。
