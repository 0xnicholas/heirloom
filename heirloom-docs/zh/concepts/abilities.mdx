---
title: 能力
description: 能力是在定义资源类型时声明的类型级能力契约。
---

**能力（Abilities）**回答智能体最重要的安全问题："我对这个实体允许做什么？"

它们在资源类型定义中声明，而不是在单独的权限表中。这使安全成为类型系统的属性，而非运行时配置。

## 能力目录

| 能力 | 含义 | 典型智能体约束 |
|---------|---------|--------------------------|
| `key` | 具有全局 RID，可被独立寻址 | 默认可用 |
| `store` | 可被嵌套在另一资源内部 | 按需 |
| `query` | 可被搜索和列表查询 | 大多数智能体持有 |
| `mutate` | 字段可被修改 | 通常不授予通用智能体 |
| `transfer` | 所有权可被转移 | 高风险操作 |
| `copy` | 可被克隆 | 按需 |
| `drop` | 可被销毁（逻辑删除） | 几乎不授予智能体 |
| `freeze` | 可被冻结为不可变状态 | 按需 |

## 核心原则

如果资源类型未声明某项能力，则**任何角色都无法执行该操作**——包括管理员角色。没有后门。

例如，如果 `Customer` 未声明 `drop`，则不可能创建删除 Customer 的操作。该操作在本体语义空间中根本不存在。

## 为什么对智能体重要

当前的智能体安全往往依赖：

- 可被绕过的提示词指令。
- 粒度粗糙的 API 密钥权限。
- 工具开发者是否记得添加权限检查。

能力将保证从人类勤勉提升到类型系统强制。如果类型系统从未允许删除，智能体就不可能意外删除数据。

## 声明能力

能力是资源类型定义的一部分：

```json
{
  "name": "Customer",
  "abilities": ["key", "query", "mutate", "freeze"],
  "fields": {
    "name": "string",
    "tier": "string"
  }
}
```

注意 `drop` 被省略。任何人——人类或智能体——要创建删除操作，都必须先通过治理流程变更资源类型。
