# ADR-006: Relationship 的三级语义（非通用边）

## 状态
Accepted

## 日期
2026-05-28

## 上下文

传统图数据库（Neo4j、TypeDB）和 Palantir Ontology 使用通用的「有向有类型的边」来表达实体间的所有关系。`Customer -[placed]-> Order` 和 `Customer -[owns]-> Address` 在系统中是同一种抽象——有向边，仅名称不同。

这种通用抽象掩盖了一个关键问题：**「删除源节点时，目标节点应该怎么办？」** 答案取决于关系的业务语义，但通用边无法编码这个语义——它需要开发者在对每个关系类型的处理代码中手动编写。

在 AI Agent 场景中，这个问题更为严重——Agent 需要理解「如果我建议删除这个客户，会有什么连锁反应」。通用边无法给出这个答案；Agent 需要依赖训练数据中的模式匹配或额外的业务规则文档。

## 决策

**引入三种具有精确语义的 Relationship，作为系统级原语：**

| 关系类型 | 生命周期依赖 | 删除行为 | 权限传播 | 所有权可转移 |
|---------|------------|---------|---------|------------|
| **Ownership** | 被拥有方依赖拥有方 | 拥有方删除 → 被拥有方级联处理 | 拥有者的 Capability 向下传播 | 是 |
| **Reference** | 被引用方不依赖引用方 | 被引用方删除 → 引用断裂（可检测）| 不传播 | 否 |
| **Association** | 双方独立 | 任意一方删除不影响另一方 | 不传播 | 否 |

**在建模时必须选择一种语义，而非留待运行时决定。**

## 后果

**积极**：
- 删除操作的级联行为由关系语义自动决定——开发者不需要手写级联逻辑（消除遗漏风险）
- Agent 可以通过查询关系的语义类型来判断操作的连锁影响——不需要额外的业务规则文档
- 权限传播被编码在关系语义中——Ownership 自动传递 Capability，不需要手动配置 Sharing 规则

**消极**：
- **语义刚性**：某些实体间的关系可能在不同上下文中具有不同语义（如 Order 和 Customer 通常是 Association，但在欺诈场景中可能需要 Ownership 级别的耦合）。三种固定语义无法表达这种上下文敏感性
- 建模者在初始阶段必须做出正确选择——如果错误地将 Ownership 建模为 Association，后续修正需要数据迁移和级联逻辑重写
- 三种语义可能在某些极端场景中不够（如「租借」——有时间限制的临时 Ownership），需要未来扩展

## 备选方案

### 方案 A：通用边 + 行为注解
所有关系使用通用边，但在边的元数据中标注「删除时对目标做什么」（CASCADE / SET_NULL / RESTRICT / NO_ACTION）。

**放弃理由**：这是 SQL 外键的模型——它解决了删除行为，但无法编码「权限是否沿关系传播」「所有权是否可以转移」等更丰富的语义。此外，SQL 外键模型的 RESTRICT 和 NO_ACTION 在语义上模棱两可。

### 方案 B：可扩展的关系语义
不固定为三种，允许用户自定义关系语义类型。

**放弃理由**：在 MVP 阶段引入自定义语义的复杂性（用户需要同时定义新语义的删除行为、权限传播规则、所有权转移规则等）超过了收益。三种基础语义覆盖了绝大多数企业场景，后续可通过 Proposal 增加新的语义类型。
