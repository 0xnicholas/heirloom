# ADR-008: Function 作为一等领域学原语

## 状态
Accepted

## 日期
2026-05-28

## 上下文

Heirloom 的动力学原语需要覆盖两种本质上不同的操作：

1. **改变世界的操作**——修改 Resource 字段、推进状态机、转移所有权（即 Action）
2. **基于世界状态的计算**——读取 Resource 属性、遍历关联、运行推导逻辑并返回结果

第二种操作是否只是一个特殊形式的查询（Query），还是需要成为一个独立的一等原语（Function）？关键问题是：**计算逻辑应该是本体系统的一部分，还是应该外置在应用程序中？**

如果计算逻辑外置，每个应用（Workshop、AI Agent、BI 工具）都需要自己实现相同的计算——例如「客户风险评分」需要在 Workshop 的视图层、Agent 的分析逻辑和 BI 的语义模型中分别实现。这不仅引入了重复，还带来了不一致的风险——同一个业务概念在不同消费端可能产生不同的计算结果。

如果计算逻辑内置为本体系统的一部分，Function 可以被所有消费端复用，且其计算过程可以被审计。

## 决策

**Function 成为一等领域学原语——与 Action 平级，但语义正交。**

Function 的定义：
- **基于代码的逻辑**：一段可执行的代码（而非声明式规则），接收对象和对象集作为输入，读取属性值，遍历关联，返回计算结果
- **只读无副作用**：不改变任何 Resource 状态
- **本体原生集成**：可接收 Resource Type 和 Object Set 作为输入参数类型，可在 Action 的 validate 和 execute 阶段被调用，可在应用程序中直接通过 API 调用
- **仅需 query Capability**：与 Action 不同，Function 不走过完整的九步流水线——它跳过 State（不改变状态）、Validate（不写入）和 Notify（不产生下游影响）。Event 可选

Function 与 Query 的区别：

| | Query（语义查询） | Function |
|------|-------------------|----------|
| 本质 | 数据检索 | 计算逻辑 |
| 输入 | JSON DSL 查询定义 | 对象 / 对象集 / 标量参数 |
| 内部逻辑 | 由 Query Resolver 翻译为底层查询计划 | 由开发者编写的代码（可包含条件、循环、多步计算） |
| 返回 | 匹配过滤条件的 Resource 集合 | 任意计算值（标量、对象、集合） |
| 可被谁调用 | 所有消费者直接使用 | Action 内部、应用、Agent 直接调用 |
| LLM 生成 | Agent 生成 DSL 查询 | Agent 可直接调用已命名的 Function——更安全 |

## 后果

**积极**：
- **单一计算来源**：同一业务逻辑（如 `risk_score()`）在所有消费端返回相同结果
- **Agent 操作简化**：Agent 不需要生成复杂的计算逻辑——它只需要知道 Function 的名称和参数，系统负责执行。这降低了 LLM 产生计算错误的概率
- **可审计的可选性**：高频 Function（如 UI 中的实时计算）可关闭审计，避免 Event Log 膨胀；关键决策中使用的 Function（如 `risk_score()` 被用于 `update_tier` 的验证）可以开启审计

**消极**：
- **代码治理负担**：Function 是代码，需要版本控制、测试和审查——比声明式的语义查询有更高的维护成本
- **性能不可预测**：Function 中的计算复杂度由开发者决定——一个遍历大型对象集并执行复杂计算的 Function 可能导致长延迟
- **本体的复杂度增加**：在 Schema Registry 中不仅需要管理类型定义，还需要管理可执行代码——模糊了类型系统和执行引擎的边界

## 备选方案

### 方案 A：计算外置（不用 Function）
所有计算逻辑留在应用程序中。Heirloom 只提供查询和 Action。

**放弃理由**：导致计算逻辑的碎片化——Agent 的计算、Workshop 的计算、BI 的计算各自独立实现。对于需要一致性的业务计算（如风险评分、合规判断），这种碎片化是不可接受的。此外，Agent 需要自行生成计算逻辑——这比调用已命名的 Function 更容易出错。

### 方案 B：用声明式规则替代 Function
用声明式规则（如 TypeDB 的 `rule` 语法）而非可执行代码来表达计算。

**放弃理由**：声明式规则适合推导新关系（如「A 是 B 的祖父」），但不适合复杂的数值计算或外部数据聚合（如「调用外部汇率 API 计算多币种总价值」）。Function 提供代码的完整灵活性，声明式规则无法覆盖这一需求。
