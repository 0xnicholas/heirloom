# ADR-037: DuckDB 作为 raw 数据分析存储层

## 状态

Proposed

## 日期

2026-07-08

## 上下文

Heirloom 的核心定位是语义层，不直接拥有客户业务数据（ADR-003）。但实际场景要求：用户需要能够快速查询和分析来自多个客户系统的 raw 数据。如果每次都实时连接源系统，会面临以下问题：

1. **性能不可控**：跨源 join、聚合分析在远程数据库上执行， latency 高且依赖源系统负载。
2. **源系统压力**：复杂分析查询可能拖慢客户生产系统。
3. **查询一致性**：不同源的查询语言、权限模型、网络可达性各异，难以统一。
4. **离线/沙箱分析**：数据团队需要在一个隔离环境中自由探索 raw 数据。

因此，在语义层和客户数据源之间需要增加一层：**raw 数据的本地物化副本 + 分析引擎**。

## 决策

**引入 DuckDB 作为 Heirloom 的 raw 数据分析存储层。**

### 定位

DuckDB 层不是替代客户数据源，而是为 Heirloom 的语义查询和分析提供高性能本地副本：

- **语义层**负责“找到数据”和“理解数据”。
- **DuckDB 层**负责“自由分析”和“快速查询”。
- **客户数据源**是权威来源，DuckDB 按需从中拉取。

### 架构关系

```
消费层（SDK / Workshop / MCP）
        │
        ▼
Semantic Layer（heirloom-core）
        │
        ▼
Query Router
  ├─ 语义查询 / 已映射字段 → 走 Mapping Engine
  └─ 下钻分析 / 原始 SQL → 走 DuckDB
        │
        ▼
DuckDB（每租户一个持久实例）
        │
        ▼
Connector / Ingestion → 客户数据源
```

### 关键设计点

1. **每租户一个 DuckDB 实例**：数据文件持久化到磁盘，租户之间完全隔离。
2. **按需同步**：用户发起分析时，先从源系统拉取所需表到 DuckDB，再执行查询。
3. **统一 DSL 下钻**：Heirloom JSON DSL 扩展下钻能力，最终翻译成 DuckDB SQL。
4. **新鲜度管理**：每次查询前检查目标表 freshness，过期则重新拉取。
5. **缓存副本而非主存储**：DuckDB 中的 raw 数据是分析缓存副本，权威数据源仍是客户系统。副本有 TTL 和清理策略，Heirloom 不主张“拥有”这些 raw 数据。
6. **持久化后端抽象**：通过 `DuckDbStorageBackend` 接口解耦 DuckDB 文件的存储位置。MVP 使用本地文件实现；未来多租户阶段可切换为对象存储实现，不影响 Ingestion 和查询逻辑。
7. **单写者约束**：DuckDB 是嵌入式 OLAP 引擎，不支持多实例并发写入同一个 `.db` 文件。架构上必须保证任一时刻只有一个写入者；读取可以多实例并发，但需先获取最新文件版本。

### 与 ADR-003 的关系

[ADR-003](./003-storage-separation.md) 强调 Heirloom 不直接“拥有”客户业务数据，资源主体应存储在客户数据源或专用 Resource Store 中。本 ADR 在 **分析缓存** 场景下引入 raw 数据副本：

- DuckDB 中的数据是**临时性分析副本**，用于加速查询和沙箱分析。
- 客户系统仍是权威数据源，DuckDB 通过同步策略保持最终一致。
- 副本生命周期由租户级策略控制（TTL、归档、清理），不成为主存储。

若未来 DuckDB 副本需要长期保留并承担写入/更新职责，应重新评估 ADR-003。

### 安全与合规

| 风险 | 缓解措施 |
|------|---------|
| 租户数据隔离 | 每租户独立 DuckDB 文件目录，文件系统权限隔离 |
| 静态数据安全 | DuckDB 文件启用加密（如 DuckDB 原生加密扩展或文件系统级加密） |
| 传输安全 | 从源系统同步数据使用 TLS；DuckDB 文件访问受服务进程权限控制 |
| 数据驻留 | 租户 DuckDB 文件存储位置可配置，满足不同区域合规要求 |
| 数据保留 | 按租户策略自动清理过期表/分区，避免无限增长 |
| 敏感数据 | Profiling 阶段识别疑似 PII，访问受 Heirloom Role/Capability 控制 |

## 后果

**积极**：
- DuckDB 单机分析性能高，支持复杂 SQL 和聚合。
- 不引入重量级分布式 OLAP 集群，运维简单。
- 租户隔离天然清晰，一个租户一个文件目录。
- 可作为 profiling、实体解析等计算的算力底座。

**消极**：
- 首次查询延迟取决于数据量和网络，需要缓存策略。
- 多租户场景下磁盘占用随租户数线性增长。
- DuckDB 是嵌入式引擎，水平扩展能力有限；超大规模租户需要重新评估。
- 需要管理数据新鲜度和租户数据生命周期（归档/清理）。

## 备选方案

### 方案 A：ClickHouse / StarRocks / Apache Doris

使用分布式列式 OLAP 引擎。

**放弃理由**：引入额外基础设施和运维复杂度。Heirloom 当前阶段以中小规模租户为主，DuckDB 的性能和部署简单性更符合 MVP 需求。

### 方案 B：PostgreSQL 作为统一存储

复用现有 PostgreSQL 存储 raw 数据副本。

**放弃理由**：PG 的分析性能（尤其是复杂聚合、宽表扫描）不如 DuckDB；且 raw store 与分析负载混合到同一数据库会增加运维风险。

### 方案 C：纯联邦查询，不保存 raw 副本

查询时实时 push-down 到各数据源。

**放弃理由**：跨源 join 和聚合性能差，无法支持离线/沙箱分析，源系统压力大。
