# ADR-003: 存储层分离（Resource Store / Graph Store / Event Log / Indexes）

## 状态
Accepted

## 日期
2026-05-28

## 上下文

Heirloom 的存储需求跨越四个截然不同的访问模式：

- **属性查询**：按 RID 点查、按类型批量扫、按属性值过滤——类似文档数据库的访问模式
- **关系遍历**：沿 Ownership / Reference / Association 路径多跳遍历——类似图数据库的访问模式
- **事件追加与范围查询**：不可变追加写入 + 时间范围扫描——类似日志存储的访问模式
- **全文/向量搜索**：关键词搜索、语义向量相似度检索——类似搜索引擎的访问模式

使用单一存储后端（如仅用 PostgreSQL JSONB 存储一切）可以简化运维，但可能在某些访问模式上产生显著的性能折损。

## 决策

**采用多模态分离存储**，将不同访问模式匹配到不同类型的存储后端：

| 存储组件 | 内容 | 参考技术 |
|---------|------|---------|
| Resource Store | Resource 主体（字段、状态、版本号） | 文档数据库（PostgreSQL JSONB / MongoDB） |
| Graph Store | Resource 间的关系图 | 属性图数据库（Neo4j / TypeDB） |
| Event Log | 不可变操作事件流 | 日志存储（Kafka 长期留存 / PostgreSQL 分区表） |
| Indexes | 属性索引、全文索引、向量索引 | Elasticsearch / pgvector |

**分离但非异质**：在 MVP 阶段，可以使用 PostgreSQL 作为统一的底层引擎，通过不同的 Schema 组织（JSONB 表、邻接表、分区表、GIN 索引）实现逻辑分离。这保留了未来切换到专用引擎的可能性。

## 后果

**积极**：
- 每个查询类型使用最优的数据结构——图遍历不需要物化关系为宽表，点查不需要走图查询引擎
- 未来可以独立水平扩展瓶颈组件（如图遍历频繁时单独扩容 Graph Store）
- 各存储组件的 schema 可以独立演化，不受其他访问模式的约束

**消极**：
- **跨存储事务一致性**：一个 Action 可能同时写入 Resource Store（字段更新）和 Graph Store（关系变更）。在没有原生分布式事务的情况下，需要应用层 Saga 或补偿机制保证最终一致性——这引入了短暂的窗口不一致
- 运维复杂度显著增加：与单一 PostgreSQL 相比，多存储（即使只用 PostgreSQL 的不同 schema）也需要更复杂的监控、备份和故障恢复策略
- 查询跨组件时需要多次往返——Query Resolver 必须编排多个存储的查询并合并结果

## 备选方案

### 方案 A：统一 PostgreSQL JSONB + 邻接表
用一个 PostgreSQL 实例存储一切：Resource 作为 JSONB 文档，关系作为邻接表行，事件作为分区表行。

**放弃理由**：图遍历（尤其是多跳深层遍历）在邻接表上的 SQL（递归 CTE）可读性极差且性能不可预测——3 跳以上的递归查询在数据量稍大时急剧退化。此外，如果未来因性能原因需要迁移到专用图存储，所有的关系查询逻辑都需要重写。从设计阶段就采用逻辑分离，可以避免未来的高成本重构。向量索引需要 pgvector 扩展也能自然地纳入 Indexes 组件的范围。

### 方案 B：Neo4j 统一存储
用 Neo4j 图数据库存储一切（属性作为节点标签，事件作为额外边或节点）。

**放弃理由**：图数据库在属性过滤和聚合方面通常不如文档数据库高效。此外，Event Log 的不可变追加模型与图数据库的优化方向（频繁更新节点和边）不匹配。
