# 摘要

**Heirloom** 是一个 AI 原生的下一代语义本体系统。它的核心命题是：当 AI Agent 进入企业运营，它们需要一个**结构化的、可被安全操作的业务语义界面**——不是数据库直连权限，不是脆弱的 prompt 约束，而是一个以 Resource 为核心的、具有类型级安全保证的本体层。

在 Heirloom 中，每一个业务实体（客户、订单、物料、合同）被建模为一个 **Resource**——它拥有明确的所有者、声明式的能力契约（Abilities）和受系统强制执行的状态机。AI Agent 像人类用户一样，通过 Role 获取 Capability，通过 Action 执行操作。当 LLM 产生幻觉说「删掉所有订单」时，不是 prompt 在阻止它——是类型系统在拒绝它。

Heirloom 的设计哲学是：**不依赖行为体的自我约束，而依赖系统的不可逾越边界。** 这使得它区别于一切将安全性寄托于「用户权限配置」或「AI 对齐训练」的系统。

本白皮书阐述 Heirloom 的设计哲学、核心概念与技术架构，与 Palantir Ontology 进行系统级设计对比，并以真实业务场景演示从语义建模到 AI 自主操作的完整闭环。附录中提供了与主流开源方案的参照分析。

# 引言

## AI Agent 进入企业：一个新的结构性危机

过去两年，企业 AI 应用经历了从「ChatGPT 问答」到「Agent 自主操作」的范式跃迁。AI 不再只是回答问题——它被期望查询数据、分析趋势、触发工作流、甚至做出操作决策。

但这种跃迁暴露了一个此前被忽视的危机：**AI Agent 在企业数据面前，只有两种同样危险的选择。**

- **选择 A —— 不给访问权限**：Agent 被限制在沙箱中，只能处理用户粘贴的数据。安全，但无用。Agent 无法发挥自主分析的价值。
- **选择 B —— 给数据库直连权限**：Agent 获得一个数据库用户的连接串，可以执行任意 SQL。有用，但极不安全。LLM 的幻觉可能被翻译为 DROP TABLE；prompt 中的「不要删除数据」只是一条建议，不是一条约束。

这不是 AI 的对齐问题。这是**架构问题**。企业缺少一个介于「无访问」和「完全访问」之间的中间层——一个 Agent 可以安全地理解、查询和操作业务数据的语义界面。这就是 Heirloom 的出发点。

## 已有尝试及其局限

**数据目录**（DataHub、Apache Atlas）告诉 Agent「数据在哪里」，但不提供对数据本身的类型安全操作。Agent 知道 `customers` 表存在，但仍然需要直接 SQL 访问它。

**图数据库**（Neo4j、TypeDB）提供了关系建模的能力，但将权限和生命周期管理留给应用层。Agent 获得了遍历关系的能力，同时也获得了「意外删除节点」的能力。

**Palantir Ontology** 最接近一个可被 Agent 使用的语义层。它通过 Action Types 约束了写入路径，使得 Agent 的操作可以被审计。但它仍将权限置于外部的 RBAC 配置中，而非类型定义内部——这使得 Agent 的安全边界依赖于管理员在另一个系统中的谨慎配置。

**Function Calling / Tool Use**（当前主流的 Agent 安全方案）通过为 Agent 提供经过封装的工具函数来限制其能力。这比直连数据库进步了一步，但安全保证仍然在应用层——每一个工具函数的开发者都必须记得加入权限检查，忘记一次就是一次安全漏洞。

## Heirloom 的定位：AI 原生的语义本体系统

Heirloom 不是另一个数据目录，不是另一个图数据库，也不是 Palantir 的开源替代品。它回答的是一个新问题：**如果 AI Agent 是企业数据的一等消费者——与人类用户平级——那么数据层需要提供什么样的基础设施？**

答案是三层：

1. **语义层**：Agent 不需要知道数据在 PostgreSQL 的哪张表、REST API 的哪个端点。Agent 只需要知道业务中有 `Customer`、`Order`、`Product`，以及它们之间的关系。语义中枢负责翻译为底层查询。

2. **类型级安全层**：一个 Resource Type 是否允许被删除、被修改、被转移——不是 RBAC 配置，是类型定义。这意味着不存在「管理员不小心给 Agent 配了一个过宽的权限」的可能——如果 `Customer` 类型没有声明 `drop`，没有任何角色能创建一个可删除 Customer 的 Action。

3. **统一校验链**：人类用户、AI Agent、自动化工作流——三者经过完全相同的 Auth → Role → Capability → Action 流水线。Agent 不会因为它是 AI 而获得更少或更多的权限——它和人类在同一个安全模型中被平等对待。

Heirloom 的设计受到类型系统理论中「使非法状态不可表达」这一思想的深刻影响。它不是试图让 Agent 变得「更听话」，而是让系统本身「使有害操作不可表达」。
